# Выгрузка архива камер key.rt.ru / camera.rt.ru

Скрипт [`rt_archive_export.py`](rt_archive_export.py) выгружает архивную запись с камеры
в обычный `.mp4`.

## Как это работает

Веб-плеер камеры отдаёт фрагментированный MP4 (fMP4) по WebSocket.
Скрипт подключается к тому же эндпоинту, что и плеер, делает «перемотку» в нужный
момент архива и сохраняет поток:

```
→ seek: <unix>     перемотка к моменту архива
→ speed: <N>       множитель скорости выгрузки
→ resume           начать отдавать видео
← fMP4 фрагменты   (ftyp+moov один раз, далее moof+mdat ~0.5 c каждый)
```

Авторизация — по тому же `access-token`, что и у основного скрипта
`../rt_key_to_go2rtc.py`. По токену скрипт сам получает свежий
`streamer_token` камеры и адрес стрим-сервера, поэтому отдельно его указывать не нужно.

## Требования

- Python 3.10+
- `pip install websockets requests`
- `ffmpeg` (нужен только для склейки, если выгрузка шла в несколько сессий)

## Время = реальное время = часы на видео

`seek`/таймстампы — это **настоящий unix-time**. Проверено: они совпадают и с
часами, вшитыми в кадр камеры, и с таймлайном плеера (плеер рисует unix-время в
таймзоне камеры, `utc_offset = 300` → UTC+5). **Никакого «сдвига» нет** — если на
видео `20:52`, то и в архиве это `20:52`.

Поэтому границы можно задавать человеческим временем (трактуется в таймзоне камеры,
UTC+5) — оно совпадёт с тем, что вшито в кадр.

## Пример (скорость 10) — основной режим

```bash
cd /home/mike/rt-key-to-go2rtc

# токен можно положить в переменную, чтобы не дублировать
export ACCESS_TOKEN='your_token'   # ваш access-token

python3 archive/rt_archive_export.py \
  --access-token "$ACCESS_TOKEN" \
  --camera <CAMERA_ID> \
  --start "2026-06-19 20:52:27" \
  --end   "2026-06-20 15:57:00" \
  --speed 10 \
  --out archive/cam_19jun2052_20jun1557.mp4
```

То же самое можно задать unix-таймстампами:

```bash
python3 archive/rt_archive_export.py \
  --access-token "$ACCESS_TOKEN" \
  --camera <CAMERA_ID> \
  --start 1781884347 \
  --end   1781953020 \
  --speed 10 \
  --out archive/cam_19jun2052_20jun1557.mp4
```

Короткий кусок через длительность вместо `--end`:

```bash
python3 archive/rt_archive_export.py \
  --access-token "$ACCESS_TOKEN" \
  --camera <CAMERA_ID> \
  --start "2026-06-19 20:52:27" \
  --duration 30m \
  --speed 10 \
  --out archive/sample.mp4
```

Длинная выгрузка в фоне с логом:

```bash
nohup python3 archive/rt_archive_export.py \
  --access-token "$ACCESS_TOKEN" \
  --camera <CAMERA_ID> \
  --start "2026-06-19 20:52:27" --end "2026-06-20 15:57:00" --speed 10 \
  --out archive/cam_19jun2052_20jun1557.mp4 > archive/export.log 2>&1 &
tail -f archive/export.log
```

## Почему скорость 10

Сервер не теряет фрагменты — он **троттлит**, то есть на большой `--speed` просто
отдаёт быстрее, дыр в архиве не возникает. Практический потолок ~16–17x.

- `--speed 10` — самый стабильный режим: соединение, как правило, **не рвётся**, вся
  запись выгружается **одной сессией** → один файл, без склейки и без дублей на стыках.
  19 ч архива ≈ ~3.5 ч реального времени.
- `--speed 16…32` — быстрее (~13–17x), но сервер периодически (примерно раз в ~6 мин)
  разрывает соединение. Скрипт сам переподключается и продолжает с места обрыва, но
  тогда выгрузка идёт **по частям** (`*.partNNN.mp4`), которые в конце склеиваются
  через ffmpeg. На стыках возможны ~3 c дубля (перемотка прыгает на ближайший keyframe).

Для многочасовых выгрузок рекомендуется `--speed 10`.

## Аргументы

| Аргумент | Назначение |
|---|---|
| `--access-token` | **(обяз.)** access-token от oauth2.key.rt.ru |
| `--camera` | **(обяз.)** id камеры (uuid) |
| `--start` | **(обяз.)** начало: unix-ts, `now`, или `"ГГГГ-ММ-ДД ЧЧ:ММ:СС"` (время камеры) |
| `--end` | конец (те же форматы, что и `--start`) |
| `--duration` | длительность, если не задан `--end`: `90s`, `15m`, `2h`, `1h30m` (по умолч. `10m`) |
| `--speed` | множитель скорости выгрузки (по умолч. `16`; рекомендуется `10`) |
| `--out` | путь к выходному mp4 (по умолч. `archive/<camera>_<start>.mp4`) |
| `--faststart` | доп. проход ffmpeg → «плоский» moov. Нужно ~2× места на диске. По умолчанию файл и так перематывается, второй проход не делается |
| `--probe` | только подключиться, показать первые кадры и выйти (диагностика) |
| `--fragment-length` | длина фрагмента на сервере, c (по умолч. `0.5`) |

## Где взять camera id и access-token

- **camera id** — поле `id` в `../cameras.json` (или из вывода `../rt_key_to_go2rtc.py`).
  Список камер с названиями:
  ```bash
  python3 -c "import json;[print(c['id'], c['title']) for c in json.load(open('cameras.json'))['data']['items']]"
  ```
  ID можно посмотреть в браузере по F12 https://key.rt.ru/main/pwa/dashboard вкладка Network, socket смотреть wss://live-vdk4.camera.rt.ru/stream/<id камеры>/live.mp4?<другие параметры>
- **access-token** — получается логином в `../rt_key_to_go2rtc.py` (`--phone`/`--password`)
  или из существующей сессии. Действует долго (месяцы). Токен можно посмотреть в браузере по F12 https://key.rt.ru/main/pwa/dashboard запрос barrier, заголовок Authorization:Brearer <token>


## Полезное

- Диск: видео ~0.9 Мбит/с → ~**0.4 ГБ в час**. 19 ч ≈ ~7.5 ГБ. При `--speed 16+` и склейке
  частей временно нужно в ~2–3 раза больше места.
- Глубина архива камеры — поле `archive_length` (в минутах) в `cameras.json`
  (у примера выше — 7 суток).
- Прогресс в выводе: текущее время архива, %, размер, фактический множитель скорости,
  число фрагментов.
- Прервать можно `Ctrl+C` — то, что успело скачаться, будет финализировано в файл.
- Проверить готовый файл:
  ```bash
  ffprobe -v error -show_entries format=duration,size -of default=noprint_wrappers=1 archive/выход.mp4
  ```

## Проигрывание

Готовый файл рекомендуется смотреть в [mpv](https://mpv.io/):

```bash
mpv archive/выход.mp4
```

Скорость воспроизведения меняется на лету прямо с клавиатуры: `]` — ускорять,
`[` — замедлять, `Backspace` — вернуть нормальную (1×).
